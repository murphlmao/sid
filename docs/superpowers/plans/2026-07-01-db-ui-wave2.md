# Database Tab UI — Wave 2, Increment 1 (connect → query → results)

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development /
> executing-plans. Steps use `- [ ]`. **Runs concurrently with the SSH split-session track
> (P3.5).** Both touch `crates/sid` incl. `app.rs` — keep `app.rs` edits localized (put the DB
> tab in its own module; touch `app.rs` only for a field + the `Tab::Database` arm + init).

**Goal:** A working Database tab: pick a saved connection, write SQL in a highlighted editor,
Run, and see rows in a grid. Built on `gpui-component` (crates.io, gpui-0.2.2) + the already-merged
`sid-db` adapter and `Store` connection facade. Backend security (rustls TLS) is already done.

**Architecture:** Adopt `gpui-component` (the validated GPUI widget lib — dbflux/pgui precedent) for
the SQL editor (tree-sitter highlighting), the results table, and text inputs. A small `DbRegistry`
(mirror the POC's `db_registry.rs`) maps `DbKind` → (`DbClient` factory, `DbClientDescriptor`). The
DB tab reads connections via `Store::read_connections`, opens the chosen one through the registry
(secret resolved from the keyring), runs `query_paged` on the shared Tokio runtime, and renders the
`QueryPage` in a `gpui-component` table. Connection add/edit uses the descriptor-driven form pattern
(same shape as the SSH host form) writing through `Store::write_connection` with save-to scope.

**Tech Stack:** Rust 2024 · gpui 0.2.2 · **gpui-component 0.5** · sid-db (`DbClient`/registry) ·
sid-core (`DbKind`, `DbClient` trait) · sid-store (connection facade) · sid-secrets (keyring).

## Global Constraints
- **Adapter pattern:** GPUI + `gpui-component` only in `crates/sid`; DB drivers only via
  `sid_db`/`sid_core::db` traits — never name `tokio-postgres`/`rusqlite` in `crates/sid`.
- **Spine:** connections persist through `Store` (attributive layered scope: global + workspace);
  secrets in the keyring by `secret_ref` (never in the DSN/config). Reuse the host-form save-to +
  secret-lifecycle pattern (mint-before-write, delete-old-after-durable-write).
- **Blocking rule:** `open`/`query_paged`/`schema_introspect` on the shared runtime
  (`terminal::ssh_runtime()` or a db runtime), never in `render`. Render pure-from-cache.
- **Pragmatic TDD (standing rule):** TDD only pure logic — the registry lookup, connection-form
  validation/DSN assembly (descriptor `assemble_params`), the query→row-model mapping if any pure
  transform exists. Editor/table/grid rendering is observation-gated. No exhaustive permutations.
- **`gpui-component` learning:** its API is non-trivial. Use context7/docs.rs for `gpui-component`,
  AND read the cloned dbflux at `<scratchpad>/dbflux` (esp. `crates/dbflux_ui_document/src/code/`
  for the editor/highlighter usage and `crates/dbflux_components/src/components/data_table/` for the
  table) as **working examples** of how to drive the widgets. Crib usage patterns, not architecture.

# Increment 2 (schema tree · cell copy/view · CSV export · query history)

**UI-only.** `DbClient::schema_introspect() -> SchemaInfo { tables: [TableInfo{ name, columns }] }`
**already exists and is implemented for Postgres, SQLite, and redb** — the tree is pure consumption,
no backend work. All of this lives in `crates/sid/src/ui/db_tab.rs`.

### D1 — Schema tree (left panel)
- On connect (and a `⟳` refresh), call `client.schema_introspect()` on the shared runtime, cache the
  `SchemaInfo`, render a left tree (table → columns) beside the editor/results. Clicking a table name
  inserts `SELECT * FROM <table>` into the editor (cheap, high-value). Pure-from-cache render.
- **TDD:** one test on any pure `SchemaInfo → tree-rows` transform (skip if it's a trivial map).

### D2 — Cell copy / view
- Result-grid cell: copy its text to the clipboard; a long cell opens a small read-only view
  (reuse the safe-text-view popover/modal pattern from `ui/session.rs`). No new backend.

### D3 — CSV export *(has a real security footgun)*
- Export the current `QueryPage` to `~/Downloads/<conn>-<n>.csv`. **Guard CSV formula injection:** any
  field beginning with `=`, `+`, `-`, `@`, tab, or CR must be prefixed with a `'` before writing, and
  fields containing `"`/`,`/newline get RFC-4180 quoting. **This is the load-bearing test** — a cell
  value `=cmd|'/C calc'!A1` must round-trip as inert text, not a live formula.
- **TDD:** the escaping function — formula-lead prefixing + quoting — with the injection case above.

### D4 — Query history (in-memory)
- A per-session ring of the last ~50 run queries (no persistence — ponytail; add a store later only if
  asked). Click an entry to reload it into the editor. Dedup consecutive duplicates.
- **TDD:** ring cap + consecutive-dedup (one small test).

### D5 — Gate
- `cargo test --workspace`, `clippy -D warnings`, `fmt --check` (real exit codes).
- **Observation (needs Murphy):** connect the demo SQLite → tree shows tables/columns → click a table
  fills the editor → Run → copy a cell → export CSV and confirm a leading-`=` cell is inert → history
  reloads a prior query.
- **Deferred to inc-3:** dedicated redb-browse UI, sortable/filterable grid, multi-statement, EXPLAIN.
